Architecture Decision Records

This page documents the intentional architectural decisions behind the Three‑Tier AWS Platform — capturing context, rationale, trade‑offs, and long‑term consequences.

ADR‑001 · Approved · 10 Apr 2026

Strict Three‑Tier Architecture with Explicit Network Boundaries

Context

Prior systems suffered from implicit trust boundaries, accidental internet exposure, and unclear tier responsibilities. A predictable, reviewable baseline architecture was required.

Decision

Adopt a strict three‑tier model with a public ingress layer, a web tier, and a private application tier. Each tier communicates only with explicitly authorized adjacent layers.

Rationale

Explicit tiering reduces cognitive load, strengthens security posture, and simplifies audits and incident reasoning.

Trade‑offs

Slightly increased infrastructure complexity and configuration overhead.

Consequences

Predictable traffic flow, reduced blast radius, and improved long‑term maintainability.

Design principle: If traffic reaches a tier, it is because the design explicitly allowed it.

ADR‑002 · Approved · 10 Apr 2026

Use EC2 Instead of Managed Compute Services

Context

Managed compute abstractions obscure networking and security boundaries, making architectural reasoning and auditing more difficult.

Decision

Deploy workloads on Amazon EC2 to preserve full visibility of network, identity, and security controls.

Rationale

EC2 enforces deliberate design decisions and removes hidden control planes that complicate reviews.

Trade‑offs

Higher operational overhead and manual scaling responsibility.

Consequences

Improved clarity, stronger fundamentals, and easier future evolution.

ADR‑003 · Approved

Full Environment Isolation via Separate Terraform State

Context

Shared state across environments increases blast radius and introduces unpredictable behavior.

Decision

Each environment uses an independent Terraform backend and lifecycle.

Rationale

Environment isolation enforces safe experimentation and predictable deployments.

Consequences

Cleaner promotion, safer rollbacks, and strong ownership boundaries.

ADR‑004 · Approved

S3 Access Restricted via VPC Gateway Endpoint

Decision

Application workloads access S3 exclusively through a VPC Gateway Endpoint. No internet routing is permitted.

Rationale

Keeps data traffic internal to AWS, reduces attack surface, and supports compliance auditing.

Trade‑offs

Increased complexity in IAM and endpoint policy configuration.

ADR‑005

Promotion‑Based CI/CD with Immutable Commits

Decision

Use commit‑based promotion from Dev to Production without branch divergence.

Rationale

Guarantees infrastructure parity and simplifies rollback and debugging.

ADR‑006

Infrastructure Drift Treated as an Incident

Decision

Terraform drift detection blocks deployment and requires explicit human intervention.

Principle: Drift represents uncertainty — not something to auto‑correct.
