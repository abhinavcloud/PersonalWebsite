Architecture Decision Records

Living documentation of intentional architectural choices for the Three‑Tier AWS Platform.

ADR‑001 — Strict Three‑Tier Architecture

Status: Approved · Date: 10 Apr 2026

Context

Prior platforms suffered from implicit trust boundaries, accidental internet exposure, and unclear tier responsibilities.

Decision

Adopt a strict three‑tier model with explicit network boundaries: public ingress, web tier, and private application tier.

Rationale

Predictable traffic flow, reduced cognitive load, and improved security posture.

Trade‑offs

Slightly higher infrastructure configuration complexity.

Consequences

Reduced blast radius and improved long‑term maintainability.

ADR‑002 — EC2 Over Managed Compute

Status: Approved · Date: 10 Apr 2026

Context

Managed compute abstractions hide networking and security behavior.

Decision

Deploy workloads directly on Amazon EC2.

Rationale

Forces explicit ownership of security, identity, and network boundaries.

Trade‑offs

Increased operational responsibility.

ADR‑003 — Environment Isolation via Terraform State

Status: Approved

Decision

Each environment maintains an independent Terraform backend.

Consequences

Safer changes, cleaner promotions, and stronger ownership boundaries.

ADR‑004 — S3 Access via VPC Gateway Endpoint

Status: Approved

Decision

S3 access is restricted to a VPC Gateway Endpoint with no internet routing.

Trade‑offs

More complex IAM and endpoint policies.

ADR‑005 — Promotion‑Based CI/CD

Status: Accepted

Promote immutable commits from development to production without branch divergence.

ADR‑006 — Infrastructure Drift as an Incident

Status: Accepted

Terraform drift blocks deployments and requires explicit human action.

Principle: Drift represents uncertainty — not automation opportunity.
